② Story Generation Flow (SSE)

Real-Time SSE Pipeline
1 · User submits form
POST /api/stories/generate with childId + themePrompt. Client enters loading state.
2 · Quota gate
API checks: global daily cap (50/day), per-user daily quota (free tier), child profile ownership.
3 · Stub row created
Story record inserted with status="generating". storyId returned to client immediately.
4 · Client opens SSE channel
GET /api/stories/:id/events — long-lived HTTP connection via EventSource. useSSE hook manages lifecycle.
5 · Gemini API call
API builds personalised prompt from child profile (name, age, interests, pronunciation). Calls Gemini 2.5 Flash Lite.
6 · Pages saved + story_complete
story_pages rows inserted. Story status → "ready". SSE emits {type:"story_complete"}. Client navigates to /story/:id.
7 · Telemetry written
events table updated: story_generated with token counts, latency, cost estimate, moderation flags.

③ Authentication & Session Flow

Auth Sequence
1
Register / Login
POST /api/auth/register or /login. Password hashed with Argon2id (memory: 64 MB, time: 3, parallelism: 4).
2
JWT issued + cookie set
HS256 JWT (7-day expiry) signed with SESSION_SECRET. Set as httpOnly, SameSite=Lax cookie. SHA-256 hash of token stored in sessions table.
3
requireAuth middleware
Every protected route: verifies JWT signature → hashes token → looks up sessions table → checks expires_at. Double validation prevents token reuse after logout.
4
Logout
DELETE session row from DB. clearCookie. Even if JWT is intercepted after logout it will fail DB lookup.
5
Session enrichment
parentId + appSessionId attached to req for telemetry. App sessions track story count and pages read per login session.

⑥ Technical Insights

PostgreSQL on Railway survives redeployments

Migrated from SQLite to Railway-managed PostgreSQL so data persists across every redeploy. node-postgres pool (max 10 connections) with async/await throughout. BIGINT timestamps avoid 32-bit overflow. Startup retry loop handles service ordering races.

SSE over WebSocket is pragmatic

Story generation is inherently one-directional (server → client). SSE requires no handshake upgrade, works through all proxies/CDNs, reconnects automatically, and is handled natively by browsers with EventSource — no library needed.

Stub-then-generate pattern decouples latency

Creating the story row before the Gemini call lets the API respond immediately with a storyId. The client opens the SSE channel independently, so network glitches during story generation don't lose the result — the client can re-connect and poll status.

ULID IDs are chronologically sortable

ULIDs encode ms-precision timestamps in the first 10 chars, making ORDER BY id equivalent to ORDER BY created_at — faster queries with no extra index on small tables. The random suffix prevents guessability (unlike sequential integers).

Token hash in sessions prevents replay after logout

Storing SHA-256(token) rather than the token itself means a DB compromise only leaks hashes. Combined with hard-delete on logout, JWTs have true revocability despite being stateless tokens — a hybrid that gets the best of both models.

Monorepo with npm workspaces is lightweight but limited

Works well for 2 packages. No shared types package or code generation between api/web — types are duplicated (e.g. Child, Story). As the project grows, a packages/shared package or generated API types (e.g. openapi-ts) would reduce drift.

Telemetry is fully in-band (PostgreSQL)

All event tracking writes to the same PostgreSQL database as the product data. Fire-and-forget pattern (floating promises with .catch) ensures telemetry failures never crash the app. At scale, events should be flushed asynchronously to a separate analytics pipeline.

Global daily cap is a blunt instrument

The 500-story/day global cap protects API cost but creates a poor UX for legitimate premium users if hit. No burst reservation for premium accounts exists yet. Consider separate premium and free-tier pools.

Gemini 2.5 Flash Lite is cost-optimised

Flash Lite is the cheapest Gemini variant, well-suited for high-volume consumer apps. The stories table records estimated_cost_usd and token counts per story, enabling per-cohort cost analysis via the /metrics endpoint.

Child personalisation baked into prompts

The LLM prompt includes child name, age, interests, vibe, and optional name pronunciation hint — enabling genuinely personalised stories at zero extra model cost compared to a generic prompt.

⑦ Design Tradeoffs

Architecture Decisions
SQLite vs PostgreSQL
Chose: PostgreSQL (Railway managed)   Trade: Requires network connection, slightly more complex local dev setup. Gains: persistent data across redeployments, proper connection pooling, standard SQL types (BIGINT, DOUBLE PRECISION), scales horizontally.
Monolith vs Microservices
Chose: Monolith Express API   Trade: All routes in one process — a bug in story generation could affect auth routes. Gains: simple deployment, no inter-service latency, easier tracing.
SSE vs WebSocket vs Polling
Chose: SSE   Trade: Browser limits to 6 concurrent SSE connections per domain (HTTP/1.1). Gains: no upgrade handshake, auto-reconnect, CDN-compatible, simpler server code.
JWT + Session Table vs Pure JWT
Chose: Hybrid   Trade: Extra DB lookup on every authenticated request. Gains: true logout revocation, ability to list active sessions, brute-force detection possible via sessions table.
Client-Side Routing vs SSR
Chose: React SPA + Vite   Trade: No SEO out of the box, blank initial HTML shell. Gains: fast HMR, simple deployment as static assets, mobile-app-like feel appropriate for a parental tool.

⑧ Risk Management

RiskLikelihoodImpactMitigation
LLM generates age-inappropriate content
Medium High
Dual moderation: prompt checked before call + output flagged. Children ages 2–12 scoped. Blocked stories tracked in events. Child-safe system prompt enforced.
PostgreSQL data loss / service outage
Low Medium
Railway manages backups and HA for the Hobby plan. API retries connection on startup (10 × 3 s). Consider enabling Railway's point-in-time recovery for production.
Gemini API cost spike / abuse
Medium High
Global 50/day hard cap limits max daily spend. Per-user quota on free tier. estimated_cost_usd tracked per story. Alerts need to be added on cost_7d threshold.
SESSION_SECRET / GEMINI_API_KEY exposure
Low High
Secrets loaded from env vars (never hardcoded). .env not committed (.gitignore). .env.example references GEMINI_API_KEY. HTTPS + httpOnly cookies prevent JS token access.
No HTTPS / SameSite=None on cookie
Low Medium
Cookie uses SameSite=Lax (blocks cross-site POSTs). Production deploy must enforce HTTPS + add Secure flag to cookie. HSTS should be configured at reverse proxy.
Single-process bottleneck on story generation
Medium Medium
Gemini call is async — event loop not blocked. SSE response streams back independently. For high concurrency, move generation to a background queue (e.g. BullMQ) with worker pool.
COPPA / GDPR compliance (children's data)
Medium High
Children's data scoped to parent accounts. No direct child accounts. Email + display_name collected. No PII sent to Gemini beyond first name + age for prompt personalisation. Privacy policy + parental consent flow needed before launch.
Stale sessions not purged
Low Low
sessions table grows indefinitely; expired rows are checked but never deleted. A nightly cleanup job (DELETE WHERE expires_at < now()) should be scheduled to prevent table bloat.
SSE connection limit (6/domain in HTTP/1.1)
Low Medium
Affects only users with multiple tabs open simultaneously. useSSE hook closes previous connection on new storyId. Resolve long-term by serving over HTTP/2 (no per-domain limit).
Already protected

Argon2id password hashing · JWT + session table revocation · httpOnly cookie · SHA-256 token hash at rest · Zod input validation on all routes · Child ownership verified per request · Content moderation dual-layer

Needs attention before launch

Add Secure flag to cookie in production · Configure HSTS · Implement COPPA/GDPR consent flow · Add nightly session cleanup job · Enable Railway point-in-time recovery · Set up cost alert on /metrics

Scale-limiting factors

PostgreSQL connection pool capped at 10 (Railway Hobby plan limit) · Single-process story generation queue · In-band telemetry competes with product queries · No image CDN (illustrations are text-only today)
